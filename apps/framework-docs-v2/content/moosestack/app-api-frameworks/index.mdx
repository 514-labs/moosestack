# Bring Your Own API Framework

MooseStack supports two complementary ways to build HTTP APIs:

- **Native Moose APIs** (fastest path): define ingestion, analytics, and trigger endpoints using Moose primitives.
- **Your framework inside the Moose Runtime**: embed a supported framework inside MooseStack via `WebApp`.
- **Client-only embedding**: keep your existing app server and use Moose as a library for ClickHouse modeling + queries.

<Callout type="info" title="Start here">
  The consolidated overview of both approaches lives in
  [Moose APIs & Web Apps](/moosestack/apis).
</Callout>

## Embed a supported framework (WebApp)

Use this when you need full framework routing/middleware but still want MooseStack to run and deploy everything together.

- [Express](/moosestack/app-api-frameworks/express)
- [Fastify](/moosestack/app-api-frameworks/fastify)
- [Koa](/moosestack/app-api-frameworks/koa)
- [Raw Node.js](/moosestack/app-api-frameworks/raw-nodejs)
- [FastAPI](/moosestack/app-api-frameworks/fastapi)

## Client-only embedding (existing apps)

If you already have a running app server, keep it where it is and use Moose as a library (no separate Moose Runtime web service).

- [Embed ClickHouse analytics in Next.js](/moosestack/getting-started/existing-app/next-js)
- For query patterns, see [Reading Data](/moosestack/olap/read-data).
