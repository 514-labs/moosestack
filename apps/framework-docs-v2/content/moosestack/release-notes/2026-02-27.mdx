**Add seedFilter configuration to OlapTable for selective data seeding** ([@phiSgr](https://github.com/phiSgr))
  Added seedFilter configuration to OlapTable that allows developers to specify limit and where clauses when seeding data from remote ClickHouse sources. This enables more precise control over which data gets seeded during development and testing, with CLI flags (--limit, --all) taking precedence over table-level configurations.

  **OlapTable with seedFilter for selective data seeding**

  ```typescript
  import { OlapTable } from "@514labs/moose-lib";
  
  // Define your data model
  interface UserEvents {
    user_id: string;
    event_type: string;
    timestamp: Date;
    revenue: number;
    country: string;
  }
  
  // Create an OLAP table with seedFilter configuration
  export const userEventsTable = new OlapTable<UserEvents>("user_events", {
    // Standard table configuration
    columns: {
      user_id: "String",
      event_type: "String", 
      timestamp: "DateTime",
      revenue: "Float64",
      country: "String"
    },
    orderBy: "timestamp",
    
    // New seedFilter configuration for selective data seeding
    seedFilter: {
      // Only seed recent high-value events from key markets
      where: "revenue > 10.0 AND country IN ('US', 'CA', 'GB') AND timestamp >= '2024-01-01'",
      // Limit to 5000 rows for faster development iterations
      limit: 5000
    }
  });
  ```

  This example shows how to configure seedFilter on an OlapTable to selectively seed data from remote ClickHouse sources. The seedFilter allows you to specify WHERE clauses and row limits for more precise control over development data, making it faster to iterate with relevant subsets of production data.

**Add projection support with `_part_offset` for optimized query performance** ([@LucioFranco](https://github.com/LucioFranco))
  Users can now define projections in their data models to optimize query performance on non-primary-key columns. Projections store data in alternative orderings within ClickHouse data parts, avoiding full table scans. This feature includes support for the special `_part_offset` column and automatic DDL generation for projection management. Templates now include projection examples showing how to define optimized data access patterns.

  **Data model with projections for optimized queries**

  ```typescript
  // models.ts - Define a data model with projections for optimized queries
  import { ClickHouseTypes, Table } from "@514labs/moose-lib";
  
  export interface UserEvent {
    userId: ClickHouseTypes.String;
    eventType: ClickHouseTypes.String;
    timestamp: ClickHouseTypes.DateTime;
    value: ClickHouseTypes.Float64;
    sessionId: ClickHouseTypes.String;
  }
  
  export default {
    table: "user_events",
    // Primary ordering by timestamp for time-series queries  
    orderBy: ["timestamp"],
    
    // Projections optimize queries on non-primary-key columns
    projections: [
      {
        // Fast queries by userId without full table scans
        name: "proj_by_user", 
        body: "SELECT _part_offset ORDER BY userId"
      },
      {
        // Optimized for time-range queries per user
        name: "proj_by_user_time",
        body: "SELECT _part_offset ORDER BY userId, timestamp" 
      },
      {
        // Pre-aggregated projection for analytics
        name: "proj_user_stats",
        body: "SELECT userId, sum(value) as total_value ORDER BY userId"
      }
    ]
  } as Table;
  ```

  This example shows how to define projections in a MooseStack data model to optimize query performance. Projections store data in alternative orderings within ClickHouse data parts, allowing fast queries on non-primary-key columns like userId without full table scans.

**Add Query Layer for type-safe SQL query building** ([@georgevanderson](https://github.com/georgevanderson))
  Introduces a new Query Layer API that provides type-safe SQL query building for ClickHouse with composable helpers, validation utilities, and MCP (Model Context Protocol) integration for IDE support. Users can now define query models with dimensions, metrics, and filters, then build queries programmatically with full type safety and automatic validation.

  **Type-safe analytics query model with validation**

  ```typescript
  import { defineQueryModel, timeDimensions, filtersFromTable, columnsFromTable } from "@514labs/moose-lib";
  import { sql } from "@514labs/moose-lib";
  
  // Assume we have a data model for user events
  interface UserEvent {
    id: string;
    user_id: string;
    event_type: string;
    timestamp: Date;
    session_id: string;
    revenue: number;
  }
  
  // Define a type-safe query model for analytics
  export const eventAnalytics = defineQueryModel({
    table: UserEventsTable,
    
    // Define dimensions - groupable fields
    dimensions: {
      eventType: { column: "event_type" },
      userId: { column: "user_id" },
      sessionId: { column: "session_id" },
      // Generate time-based dimensions automatically
      ...timeDimensions(UserEventsTable.columns.timestamp),
    },
    
    // Define metrics - aggregated calculations
    metrics: {
      totalEvents: { 
        agg: sql`count(*)`, 
        as: "total_events" 
      },
      uniqueUsers: { 
        agg: sql`countDistinct(user_id)`, 
        as: "unique_users" 
      },
      totalRevenue: { 
        agg: sql`sum(revenue)`, 
        as: "total_revenue" 
      },
    },
    
    // Define filterable fields with type-safe operators
    filters: {
      ...filtersFromTable(UserEventsTable, { 
        include: ["event_type", "user_id", "timestamp"] 
      }),
      eventType: { 
        column: "event_type", 
        operators: ["eq", "in", "ne"] as const,
        inputType: "select" 
      },
    },
    
    // Columns available for selection
    columns: {
      ...columnsFromTable(UserEventsTable),
    },
    
    // Define sortable fields
    sortable: ["total_events", "unique_users", "timestamp"] as const,
  });
  
  // Use the query model in an API endpoint
  export async function getEventStats(request: unknown) {
    // Type-safe query execution with validation
    return await eventAnalytics.fromObject(request);
  }
  ```

  This example shows how to use the new Query Layer to define a type-safe analytics model for user events. The model includes dimensions for grouping data, metrics for aggregations, and filters with validation, all with full TypeScript type safety and automatic schema generation.

**Add migration configuration to ignore LowCardinality(String) vs String differences** ([@hh2110](https://github.com/hh2110))
  Added a new migration configuration option that allows users to ignore differences between LowCardinality(String) and String column types in ClickHouse tables. This prevents unnecessary migrations when the semantic meaning of the data is the same, reducing migration noise and improving developer experience when working with ClickHouse optimizations.

  **Configure migration to ignore LowCardinality differences**

  ```typescript
  // moose.config.ts - Configure migration to ignore LowCardinality differences
  import { MooseConfig } from '@moosestack/framework';
  
  export default {
    // Migration configuration to reduce noise from ClickHouse optimizations
    migrationConfig: {
      ignoreOperations: [
        // Ignore differences between LowCardinality(String) and String types
        // This prevents unnecessary migrations when ClickHouse auto-optimizes columns
        'IgnoreStringLowCardinalityDifferences'
      ]
    },
    
    // Other configuration options
    clickhouse: {
      host: process.env.CLICKHOUSE_HOST || 'localhost',
      port: 8123
    }
  } satisfies MooseConfig;
  
  // Example data model that might trigger the optimization
  // models/UserActivity.ts
  export interface UserActivity {
    userId: string;          // May become LowCardinality(String) in ClickHouse
    eventType: string;       // May become LowCardinality(String) in ClickHouse  
    timestamp: Date;
    sessionId: string;
    metadata?: Record<string, any>;
  }
  
  // With the ignore configuration above, migrations won't be triggered
  // when ClickHouse automatically converts String to LowCardinality(String)
  // for columns with low cardinality values like eventType
  ```

  Shows how to configure Moose to ignore differences between LowCardinality(String) and String column types in ClickHouse migrations. This prevents unnecessary migration noise when ClickHouse automatically optimizes string columns with low cardinality values.

**Add Merge table engine support** ([@DatGuyJonathan](https://github.com/DatGuyJonathan))
  Users can now define Merge engine tables in their data models to create unified views over multiple source tables with matching schemas. The Merge engine allows querying data from multiple tables as if they were a single table, useful for partitioned data scenarios and historical data archiving patterns.

  **Merge Engine Table for Unified Data Views**

  ```typescript
  // app/datamodels/orders.ts
  import { Key } from "@514labs/moose-lib";
  
  // Source tables with matching schemas for partitioned data
  export interface DailyOrders {
    id: Key<string>;
    customerId: string;
    amount: number;
    orderDate: Date;
    status: string;
  }
  
  export interface WeeklyOrders {
    id: Key<string>;
    customerId: string;
    amount: number;
    orderDate: Date;
    status: string;
  }
  
  // Merge table that unifies all order tables for querying
  export interface AllOrders {
    id: Key<string>;
    customerId: string;
    amount: number;
    orderDate: Date;
    status: string;
    
    // Configure Merge engine to query across multiple tables
    engine: {
      Merge: {
        sourceDatabase: "currentDatabase()", // Resolves to current database
        tablesRegexp: "^(Daily|Weekly)Orders$" // Match DailyOrders and WeeklyOrders
      }
    };
  }
  ```

  This example shows how to define a Merge engine table that creates a unified view over multiple source tables with matching schemas. The Merge engine allows querying partitioned data (DailyOrders, WeeklyOrders) as a single table (AllOrders), useful for time-based partitioning patterns.
