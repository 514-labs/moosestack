  ```bash
  # Navigate to the project directory

  # The setup command automatically:
  # - Clones the repository
  # - Creates the specified branch

  # - Sets up deployment configuration

  git add .

  cd /path/to/existing/repo
  boreal link --project acme/marketing-dashboard --yes
  
  # Or use interactive mode to select from available projects
  boreal link
  
  # Check deployment status in Boreal dashboard
  echo "View deployment: https://fiveonefour.boreal.cloud/acme/analytics-api/deployments"
  ```

  This example shows the complete workflow for using the new Boreal CLI commands to set up new projects and link existing repositories. The setup command handles cloning, branch creation, and deployment configuration, while the link command connects existing local repos to Boreal projects for streamlined deployments.

**Add lifecycle management support to materialized views** (Dave Seleno)
  Materialized views now support lifecycle management with three options: FULLY_MANAGED (default - Moose manages creation, updates, and deletion), DELETION_PROTECTED (prevents accidental deletion and updates), and EXTERNALLY_MANAGED (user manages the view outside of Moose). This gives users fine-grained control over how their materialized views are managed in production environments.

  **Materialized View Lifecycle Management Options**

  ```typescript
  import { MaterializedView, LifeCycle, Key, DateTime } from "@514labs/moose-lib";
  import { UserActivity } from "../ingest/models";
  
  // Define the aggregated data structure
  interface UserActivitySummary {
    userId: Key<string>;
    totalEvents: number;
    lastActivity: DateTime;
  }
  
  const userActivityTable = UserActivity.table!;
  
  // FULLY_MANAGED (default) - Moose handles everything
  export const userSummaryMV = new MaterializedView<UserActivitySummary>({
    materializedViewName: "UserActivitySummary",
    targetTable: {
      name: "UserActivitySummaryTable",
      orderByFields: ["userId", "lastActivity"],
    },
    selectStatement: `
      SELECT 
        userId,
        count() as totalEvents,
        max(timestamp) as lastActivity
      FROM \`${userActivityTable.name}\`
      GROUP BY userId
    `,
    selectTables: [userActivityTable],
    // lifeCycle defaults to FULLY_MANAGED - Moose creates, updates, and deletes
  });
  
  // DELETION_PROTECTED - Prevents accidental deletion or updates
  export const criticalReportMV = new MaterializedView<UserActivitySummary>({
    materializedViewName: "CriticalUserReport", 
    targetTable: {
      name: "CriticalUserReportTable",
      orderByFields: ["userId"],
    },
    selectStatement: `
      SELECT userId, count() as totalEvents, max(timestamp) as lastActivity
      FROM \`${userActivityTable.name}\`
      WHERE userId IN (SELECT userId FROM premium_users)
      GROUP BY userId
    `,
    selectTables: [userActivityTable],
    lifeCycle: LifeCycle.DELETION_PROTECTED, // Prevents drops/updates
  });
  
  // EXTERNALLY_MANAGED - User manages the view outside of Moose
  export const externalReportMV = new MaterializedView<UserActivitySummary>({
    materializedViewName: "ExternalUserReport",
    targetTable: {
      name: "ExternalUserReportTable", 
      orderByFields: ["userId"],
    },
    selectStatement: `SELECT userId, 0 as totalEvents, now() as lastActivity FROM users`,
    selectTables: [userActivityTable],
    lifeCycle: LifeCycle.EXTERNALLY_MANAGED, // Moose won't create or manage this view
  });
  ```

  This example shows how to use the three lifecycle management options for materialized views: FULLY_MANAGED (default) for automatic management, DELETION_PROTECTED to prevent accidental changes to critical views, and EXTERNALLY_MANAGED for views managed outside of Moose.

**Add external table mirroring in dev mode** (Nicolas Joseph)
  Developers can now automatically create local mirror tables for externally managed tables during development. When enabled via `create_local_mirrors = true` in moose.config.toml, the dev server will mirror remote table schemas and optionally sample data to local ClickHouse, making it easier to develop and test with external data sources without requiring direct access to production systems.

  **Enable external table mirroring in moose.config.toml**

  ```toml
  # moose.config.toml
  [project]
  name = "my-analytics-app"
  
  # ClickHouse configuration for local development
  [clickhouse]
  host = "localhost"
  port = 9000
  db_name = "analytics_dev"
  
  # Development mode configuration
  [dev.externally_managed.tables]
  # Enable automatic creation of local mirror tables for external tables
  create_local_mirrors = true
  
  # Number of sample rows to copy from remote tables (0 = schema only)
  sample_size = 1000
  
  # Whether to refresh mirrors on startup (drop and recreate)
  refresh_on_startup = true
  
  # Optional: Remote ClickHouse connection for mirroring
  [clickhouse.remote]
  url = "https://production-clickhouse.company.com:8443"
  # username and password stored securely via keyring
  ```

  This configuration enables external table mirroring in development mode. When enabled, the dev server automatically creates local mirror tables for any EXTERNALLY_MANAGED tables, optionally copying sample data from production systems to make local development easier.

**Add generate dockerfile command for Docker customization** (Mitchell Suzuki)
  Added new `moose generate dockerfile` command that creates a customizable Dockerfile without building Docker images. This allows developers to generate and modify the Dockerfile for their deployment needs before running the build process. The command requires `custom_dockerfile = true` in moose.config.toml and places the Dockerfile at the configured path (default: ./Dockerfile).

  **Generate and customize Dockerfile for Moose deployment**

  ```bash
  # First, enable custom dockerfile mode in your moose.config.toml
  cat >> moose.config.toml << EOF
  [docker_config]
  custom_dockerfile = true
  dockerfile_path = "./Dockerfile"
  EOF
  
  # Generate the Dockerfile without building images
  moose generate dockerfile
  
  # Now you can customize the generated Dockerfile
  # Add custom dependencies, caching layers, etc.
  cat >> Dockerfile << EOF
  
  # Add custom dependencies
  RUN apt-get update && apt-get install -y \\
      curl \\
      jq \\
      && rm -rf /var/lib/apt/lists/*
  
  # Add custom environment variables
  ENV CUSTOM_CONFIG_PATH=/app/config
  EOF
  
  # Build the Docker image with your customizations
  moose build --docker
  
  # Or build for specific architecture
  moose build --docker --amd64
  ```

  This example shows how to use the new generate dockerfile command to create a customizable Dockerfile for a Moose data pipeline project. The workflow demonstrates enabling custom dockerfile mode, generating the base Dockerfile, customizing it with additional dependencies, and building the final Docker image.
