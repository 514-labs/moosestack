# Row-Level Security

Row-Level Security (RLS) lets you restrict which rows a query returns based on the caller's identity. Moose implements this using [ClickHouse row policies](https://clickhouse.com/docs/en/sql-reference/statements/create/row-policy) that read JWT claim values at query time via `getSetting()`.

## How It Works

1. You define a `SelectRowPolicy` that maps a table column to a JWT claim
2. Moose generates a `CREATE ROW POLICY` DDL that filters rows where the column matches the claim value
3. At request time, Moose extracts the claim from the JWT and injects it as a ClickHouse custom setting
4. ClickHouse evaluates the row policy on every query, returning only matching rows

<Callout type="info" title="Automatic auth enforcement">
When any row policy is defined, JWT authentication is automatically required on all consumption APIs — even if `enforce_on_all_consumptions_apis` is `false`. This prevents accidental data leaks.
</Callout>

## Defining a Row Policy

Export a `SelectRowPolicy` from any file in your Moose project:

<LanguageTabs>
  <LanguageTabContent value="typescript">
```ts filename="app/rowPolicies.ts" copy
import { SelectRowPolicy } from "@514labs/moose-lib";
import { orders } from "./Orders";
import { invoices } from "./Invoices";

export const tenantIsolation = new SelectRowPolicy("tenant_isolation", {
  tables: [orders, invoices],
  column: "org_id",
  claim: "org_id",
});
```
  </LanguageTabContent>
  <LanguageTabContent value="python">
<Callout type="warning">
Row-Level Security is currently available in TypeScript only. Python support is planned for a future release.
</Callout>
  </LanguageTabContent>
</LanguageTabs>

### Configuration Reference

| Parameter | Type | Description |
|:----------|:-----|:------------|
| `name` | `string` | Unique name for the policy (used in ClickHouse DDL) |
| `tables` | `OlapTable[]` | Tables the policy applies to (ClickHouse row policies only support tables, not views) |
| `column` | `string` | Column to filter on (must exist on every listed table) |
| `claim` | `string` | JWT claim name whose value is compared against the column |

### Generated DDL

For the example above, Moose generates:

```sql
CREATE ROLE IF NOT EXISTS moose_rls_role;

CREATE ROW POLICY IF NOT EXISTS `tenant_isolation_on_orders`
  ON `local`.`orders_1_0_0`
  USING org_id = getSetting('SQL_moose_rls_org_id')
  TO moose_rls_role;

CREATE ROW POLICY IF NOT EXISTS `tenant_isolation_on_invoices`
  ON `local`.`invoices_1_0_0`
  USING org_id = getSetting('SQL_moose_rls_org_id')
  TO moose_rls_role;
```

## Querying with Row Policies

### Consumption APIs

In consumption API handlers, row policies are enforced automatically when a valid JWT is present. The JWT claim value is extracted and injected into every query.

```ts filename="app/apis/orders.ts" copy
import { ConsumptionApi } from "@514labs/moose-lib";

export const ordersApi = new ConsumptionApi("orders", async (params, { client, sql, jwt }) => {
  // Row policies are applied automatically — this query only returns
  // rows where org_id matches the JWT's org_id claim
  const result = await client.query.execute(
    sql`SELECT * FROM orders`
  );
  return result;
});
```

### Standalone / Server-Side

When using `getMooseUtils()` outside of a consumption API (e.g., in a script or custom server), pass `rlsContext` to scope queries:

```ts filename="app/server.ts" copy
import { getMooseUtils } from "@514labs/moose-lib";

const orgId = "acme-corp"; // from your auth layer

const { client, sql } = await getMooseUtils({
  rlsContext: { org_id: orgId },
});

// Queries through this client are filtered by org_id
const result = await client.query.execute(
  sql`SELECT * FROM orders`
);
```

<Callout type="warning" title="Missing claims return 403">
If a required JWT claim is missing, the API returns `403 Forbidden`. Ensure your JWT tokens include all claims referenced by your row policies.
</Callout>

## Build-Time Validation

Moose validates row policies at build time:

- **Column existence**: The `column` must exist on every table listed in `tables`
- **Claim consistency**: If multiple policies filter on the same column, they must map to the same JWT claim
- **Table existence**: Every table in `tables` must be a defined Moose primitive

Validation errors are reported before any DDL is executed, preventing invalid policies from reaching ClickHouse.

## Multiple Policies

You can define multiple row policies for different dimensions:

```ts filename="app/rowPolicies.ts" copy
import { SelectRowPolicy } from "@514labs/moose-lib";
import { orders } from "./Orders";

export const tenantIsolation = new SelectRowPolicy("tenant_isolation", {
  tables: [orders],
  column: "org_id",
  claim: "org_id",
});

export const regionFilter = new SelectRowPolicy("region_filter", {
  tables: [orders],
  column: "region",
  claim: "region",
});
```

When multiple policies apply to the same table, ClickHouse enforces all of them — a row must satisfy every policy to be returned.
