## Overview

Data is in ClickHouse. Now you need to expose it — and who's consuming it determines how.

Maybe your product team needs a dashboard showing attendance trends. Maybe you're building a chat feature where users ask questions in plain English. Maybe your engineering team wants to query production data from Claude Code without writing SQL.

These are different problems. But the metrics are the same. "No-shows" should mean the same thing whether it's rendered in a chart, returned by an AI, or discovered by an MCP client.

Moose's semantic layer lets you define your metrics, dimensions, and filters once — then project them into whatever interface the consumer needs.

## Define a query model

The [data model](/moosestack/olap/model-table) describes what's *in* the table. The query model describes what you can *ask* it. It's a structured declaration of every way a consumer is allowed to slice this data — and it's the single object that every downstream interface reads from.

<LanguageTabs>
  <LanguageTabContent value="typescript">
```ts filename="query-layer/visits-model.ts" copy
import {
  defineQueryModel, sql, count, countDistinct, sum, timeDimensions,
} from "@514labs/moose-lib";
import { VisitsTable } from "../datamodels/visits";

export const visitMetrics = defineQueryModel({
  // Name becomes the tool name in MCP and AI SDK registrations
  name: "query_visits",
  // Description becomes the tool description LLMs read to decide when to call it
  description:
    "Visit and attendance metrics: bookings, check-ins, no-shows, " +
    "cancellations. Always requires a studio_id.",

  table: VisitsTable,

  // Columns consumers can GROUP BY
  dimensions: {
    status:     { column: "status" },
    locationId: { column: "location_id" },
    // timeDimensions generates day, month, week from a date column
    ...timeDimensions(VisitsTable.columns.start_date),
  },

  // Aggregate computations
  metrics: {
    totalVisits:   { agg: count(),                                           as: "totalVisits" },
    uniqueClients: { agg: countDistinct(VisitsTable.columns.user_id),        as: "uniqueClients" },
    attended:      { agg: sum(sql`CASE WHEN status = 1 THEN 1 ELSE 0 END`), as: "attended" },
    noShows:       { agg: sum(sql`CASE WHEN status = 2 THEN 1 ELSE 0 END`), as: "noShows" },
    lateCancels:   { agg: sum(sql`CASE WHEN status = 3 THEN 1 ELSE 0 END`), as: "lateCancels" },
  },

  // Which columns can be filtered and with which operators
  filters: {
    studioId:  { column: "studio_id",  operators: ["eq"] as const, required: true },
    startDate: { column: "start_date", operators: ["gte", "lte"] as const },
    status:    { column: "status",     operators: ["eq", "in"] as const },
  },

  // Whitelist of fields that can appear in ORDER BY
  sortable: ["totalVisits", "uniqueClients", "day", "month", "week"] as const,

  // What happens when a consumer doesn't specify something
  defaults: {
    dimensions: ["day"],
    metrics: ["totalVisits"],
    orderBy: [["day", "DESC"]],
    limit: 100,
    maxLimit: 1000,
  },
});
```

/** A booking or attendance record for a class at a fitness studio. */
interface Visit {
  id: Key<number>;
  /** Tenant identifier. Every query must filter on this. */

  user_id: number;

  start_date: Date;
  created_at: Date;
}

Note how [column comments](/moosestack/olap/model-table#column-comments) (TSDoc in TypeScript, `Field(description=...)` in Python) become ClickHouse `COMMENT` clauses. These comments also surface in Moose's MCP tools — AI agents read them to understand what each column means before writing queries.
</ToggleBlock>

### Query model anatomy

Each section of the query model controls a different aspect of what consumers can do:

| Section | Purpose | Effect |
|---|---|---|
| **`name`** | Tool name for MCP and AI SDK | Becomes the callable tool name |
| **`description`** | What this model queries | LLMs read this to decide when to call the tool |
| **`table`** | Source OlapTable | The ClickHouse table queries run against |
| **`dimensions`** | Allowed `GROUP BY` columns | `timeDimensions()` generates `day`, `month`, `week` from a date column |
| **`metrics`** | Aggregate computations | Appear in `SELECT` alongside requested dimensions |
| **`filters`** | Allowed `WHERE` clauses with operators | `required: true` enforces mandatory filtering (e.g., tenant isolation) |
| **`sortable`** | Allowed `ORDER BY` fields | Whitelist — consumers can't sort by arbitrary expressions |
| **`defaults`** | Fallback values | Applied when a consumer omits dimensions, metrics, limit, or ordering |

Now: three different consumers, three different interfaces, same model.

## REST API: power dashboards and integrations

Your frontend team needs a stable endpoint they can poll. The ops team wants a weekly report they can automate. A partner wants to pull data into their own system.

These consumers know exactly what they want. They'll construct the request themselves. They need a URL, query parameters, and a predictable response shape.

The `buildQuery` builder takes the query model and gives you a type-safe fluent API. Wire it into a route handler — the builder validates dimensions, metrics, and filter operators at compile time and generates the ClickHouse SQL at runtime. `.filter()` skips `undefined` values, so optional query params work without branching.

<LanguageTabs>
  <LanguageTabContent value="typescript">
```ts filename="apis/visits.ts" copy
import { Api, buildQuery } from "@514labs/moose-lib";
import { visitMetrics } from "../query-layer/visits-model";

interface VisitQueryParams {
  studio_id: number;
  dimensions?: string;
  metrics?: string;
  start_date_gte?: string;
  start_date_lte?: string;
  limit?: number;
}

export default new Api<VisitQueryParams, any>("visits",
  async ({ studio_id, dimensions, metrics, start_date_gte, start_date_lte, limit }, { client }) => {
    const results = await buildQuery(visitMetrics)
      .dimensions(dimensions ? dimensions.split(",") : ["day"])
      .metrics(metrics ? metrics.split(",") : ["totalVisits"])
      .filter("studioId", "eq", studio_id)
      .filter("startDate", "gte", start_date_gte)
      .filter("startDate", "lte", start_date_lte)
      .limit(limit || 100)
      .execute(client);

    return results;
  }
);
```
  </LanguageTabContent>
  <LanguageTabContent value="python">
```python filename="apis/visits.py" copy
from moose_lib import Api, MooseClient, build_query
from pydantic import BaseModel
from typing import Optional
from app.query_layer.visits_model import visit_metrics

class VisitQueryParams(BaseModel):
    studio_id: int
    dimensions: Optional[str] = None
    metrics: Optional[str] = None
    start_date_gte: Optional[str] = None
    start_date_lte: Optional[str] = None
    limit: Optional[int] = None

def run(client: MooseClient, params: VisitQueryParams) -> list[dict]:
    results = (
        build_query(visit_metrics)
        .dimensions(params.dimensions.split(",") if params.dimensions else ["day"])
        .metrics(params.metrics.split(",") if params.metrics else ["totalVisits"])
        .filter("studioId", "eq", params.studio_id)
        .filter("startDate", "gte", params.start_date_gte)
        .filter("startDate", "lte", params.start_date_lte)
        .limit(params.limit or 100)
        .execute(client)
    )
    return results

visits_api = Api[VisitQueryParams, dict](name="visits", query_function=run)
```
  </LanguageTabContent>
</LanguageTabs>

A request like:

```
GET /api/visits?studio_id=5&dimensions=day&metrics=noShows&start_date_gte=2025-01-01
```

Produces this ClickHouse query:

```sql
SELECT
  toDate(visits.start_date) AS day,
  sum(CASE WHEN status = 2 THEN 1 ELSE 0 END) AS noShows
FROM visits
WHERE visits.studio_id = 5
  AND visits.start_date >= '2025-01-01'
GROUP BY toDate(visits.start_date)
ORDER BY day DESC
LIMIT 100
```

<Callout type="info" title="When to use this">
Use the REST API when the consumer is **code** — dashboards that refresh on a timer, cron jobs that generate PDFs, integrations that need to work the same way in six months. HTTP gives you caching, rate limiting, auth middleware, and all the infrastructure you already have.
</Callout>

## AI SDK tool calls: build a chat feature

You're shipping a product — a "chat with your data" feature, a copilot, an embedded analyst. The AI needs to query ClickHouse, but you need to control the experience: the system prompt, how results are rendered, what guardrails are in place.

`createModelTool` reads the query model and generates two things:
- A flat **Zod schema** — filters become `start_date_gte`, `status_in`; dimensions and metrics become enum arrays
- A **`buildRequest`** function that reverses the flattening back into a nested query request

The Vercel AI SDK's `tool()` takes a Zod schema and an execute function, so the two plug together directly.

<LanguageTabs>
  <LanguageTabContent value="typescript">
```ts filename="ai/tools.ts" copy
import { createModelTool } from "@514labs/moose-lib";
import { tool } from "ai";
import { visitMetrics } from "../query-layer/visits-model";

// createModelTool derives the Zod schema from the query model's
// dimensions, metrics, and filters
const { schema, buildRequest } = createModelTool(visitMetrics);

export const queryVisitsTool = tool({
  description: visitMetrics.description,
  parameters: schema,
  execute: async (params) => {
    const request = buildRequest(params);
    const sql = visitMetrics.toSql(request);
    const results = await client.query(sql);
    return results;
  },
});
```
  </LanguageTabContent>
  <LanguageTabContent value="python">
```python filename="ai/tools.py" copy
from moose_lib import create_model_tool
from app.query_layer.visits_model import visit_metrics

# create_model_tool derives the schema from the query model's
# dimensions, metrics, and filters
model_tool = create_model_tool(visit_metrics)

# Use with your AI framework of choice
# The schema and build_request function plug into any tool-calling interface
schema = model_tool.schema
build_request = model_tool.build_request

async def execute_query(params: dict) -> list[dict]:
    request = build_request(params)
    query = visit_metrics.to_sql(request)
    results = await client.query(query)
    return results
```
  </LanguageTabContent>
</LanguageTabs>

Then wire the tool into a chat route. This is where you add what `createModelTool` can't give you — the system prompt with domain context, the model choice, the streaming behavior:

# The query model handles *what* can be queried;
# you handle *how* it feels

User asks "how many no-shows did studio 5 have last week?" — the model calls:

```json
{
  "studio_id": 5,
  "dimensions": ["day"],
  "metrics": ["noShows"],
  "start_date_gte": "2025-02-17",
  "start_date_lte": "2025-02-23"
}
```

Same metric definition. Same generated SQL. But because the tool lives inside your app, you decide:

- **The system prompt** — add domain context so the model maps "no-shows" to the right parameter
- **The rendering** — table, chart, natural-language summary, or all three
- **The guardrails** — `studio_id` is `required: true` in the model, so the model can't query across tenants
- **The composition** — give the model multiple tools and let it chain them

<Callout type="info" title="When to use this">
Use AI SDK tool calls when you're building a **product** — you own the UX and need full control over the system prompt, rendering, and error handling. The tradeoff is that you build and maintain the integration, but you decide exactly what the experience feels like.
</Callout>

## MCP server: let everyone self-serve

Your data engineer uses Claude Code. Your PM uses Claude Desktop. Your analyst uses Cursor. They all want to ask questions about visit data without writing SQL.

You don't want to build a UI for each of them. You want to deploy one endpoint and let every client discover what's available.

`registerModelTools` does the wiring for you. It calls `createModelTool` internally for each query model, registers the resulting tools on an MCP server, and handles the parameter flattening and request building. You provide the models and a callback that executes the query. Everything else — tool names, descriptions, parameter schemas — is derived from the model definitions.

<LanguageTabs>
  <LanguageTabContent value="typescript">
```ts filename="apis/mcp.ts" copy
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StreamableHTTPServerTransport } from "@modelcontextprotocol/sdk/server/streamableHttp.js";
import { registerModelTools } from "@514labs/moose-lib";
import { visitMetrics } from "../query-layer/visits-model";
import { revenueMetrics } from "../query-layer/revenue-model";
import { clientMetrics } from "../query-layer/client-model";

export default async function handler(req, res) {
  const server = new McpServer({
    name: "my-analytics",
    version: "1.0.0",
  });

  // One call registers all models as MCP tools
  registerModelTools(
    server,
    [visitMetrics, revenueMetrics, clientMetrics],
    async (model, request, limit) => {
      const sql = model.toSql({ ...request, limit });
      const rows = await client.query(sql);
      return {
        content: [{ type: "text", text: JSON.stringify(rows) }],
      };
    },
  );

  const transport = new StreamableHTTPServerTransport({
    sessionIdGenerator: undefined,
  });
  await server.connect(transport);
  await transport.handleRequest(req, res, req.body);
}
```
  </LanguageTabContent>
  <LanguageTabContent value="python">
```python filename="apis/mcp.py" copy
from moose_lib import register_model_tools
from mcp.server import Server
from app.query_layer.visits_model import visit_metrics
from app.query_layer.revenue_model import revenue_metrics
from app.query_layer.client_model import client_metrics

server = Server("my-analytics")

# One call registers all models as MCP tools
register_model_tools(
    server,
    [visit_metrics, revenue_metrics, client_metrics],
    execute_callback=lambda model, request, limit: {
        "content": [{"type": "text", "text": str(
            client.query(model.to_sql({**request, "limit": limit}))
        )}]
    },
)
```
  </LanguageTabContent>
</LanguageTabs>

Three models, three MCP tools, one function call. Any MCP client connects and discovers the tools:

```bash copy
claude mcp add --transport http analytics https://my-app.com/mcp
```

> "Show me no-shows by week for studio 5 in January"

Every MCP-compatible client — Claude Desktop, Cursor, Claude Code, custom agents — connects to the same endpoint and discovers the same tools with the same schemas.

<Callout type="info" title="When to use this">
Use MCP when you're building **infrastructure**, not a product. You don't control the UI, the system prompt, or which model the client uses. You just make the data available and let each consumer use it however they want. The tradeoff is that you give up control over the experience.
</Callout>

## Same model, every interface

```
                              +----------------+
                              |  Query Model   |
                              |                |
                              |  dimensions    |
                              |  metrics       |
                              |  filters       |
                              |  defaults      |
                              +-------+--------+
                                      |
               +----------------------+----------------------+
               |                      |                      |
               v                      v                      v
        +-----------+          +-----------+          +-----------+
        | REST API  |          |  AI SDK   |          |    MCP    |
        |           |          | tool calls|          |   server  |
        | buildQuery|          | createModel          | register  |
        |           |          | Tool      |          | ModelTools|
        +-----+-----+         +-----+-----+          +-----+-----+
              |                      |                      |
              v                      v                      v
        Dashboards             Your chat UI           Claude Desktop
        Reports                Copilots               Cursor
        Integrations           Embedded analysts      Claude Code
```

| Goal | Interface | You control the UX? | Best when... |
|---|---|---|---|
| Power a dashboard or report | REST API via `buildQuery` | N/A | The consumer is code that knows exactly what it wants |
| Build a chat feature | AI SDK via `createModelTool` | Yes | You're shipping a product and need to own the experience |
| Let everyone self-serve | MCP via `registerModelTools` | No | You want wide reach without building custom UIs |

They compose. Use REST for the ops dashboard, embedded tools for the customer-facing chat, MCP for internal self-serve. All three read from the same query model.
